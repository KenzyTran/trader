## Why Gateway is needed

Locally, tools call Massive, Tavily, and Telegram directly. In the AWS architecture, AgentCore Gateway creates a managed MCP endpoint with Lambda tools behind it. AgentCore Identity keeps API keys outside the source code.

### Create credential providers

From `deploy/agentcore/TraderFloor`:

```powershell
agentcore add credential --name massive-api --api-key $env:MASSIVE_API_KEY
agentcore add credential --name tavily-api --api-key $env:TAVILY_API_KEY
agentcore add credential --name telegram-token --api-key $env:TELEGRAM_BOT_TOKEN
```

Sensitive values are stored in `agentcore/.env.local` during development and managed by Identity after deployment.

### Create the Gateway

For internal connectivity testing, the workshop can begin without an inbound authorizer:

```powershell
agentcore add gateway `
  --name TraderToolsGateway `
  --authorizer-type NONE `
  --runtimes TraderFloor
```

After the Lambda tool and `tools.json` are ready:

```powershell
agentcore add gateway-target `
  --name TradingTools `
  --type lambda-function-arn `
  --lambda-arn <TRADING_TOOL_LAMBDA_ARN> `
  --tool-schema-file tools.json `
  --gateway TraderToolsGateway

agentcore deploy
```

### Production hardening

- Change the Gateway to `CUSTOM_JWT` with a Cognito discovery URL and app-client audience.
- Scope the Runtime role to only permitted credential-provider ARNs.
- Separate Massive, Tavily, and Telegram credentials so they can be rotated independently.
- Never return API keys in tool responses or logs.

{{% notice warning %}}
`authorizer-type NONE` is only for experimentation. Do not use it for an externally accessible endpoint.
{{% /notice %}}
